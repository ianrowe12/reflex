type: visual
title: "Data Flow — Happy Path"
subtitle: "From sensor change to recommendation delivered"
bullets:
- "End-to-end: ~2-5 minutes from sensor shift to supervisor notification"
- "Numbers are NEVER generated by the LLM — programmatically extracted and cross-validated"
- "Deterministic template fallback if Claude is down or validation fails"
speaker_notes: "Walk through this left to right. Step 1: PI Historian sends sensor readings to our Edge Agent in the refinery DMZ. The Edge Agent does a quick quality pre-check and buffers to SQLite in case cloud connectivity drops. Step 2: validated data pushes outbound-only over HTTPS to Azure IoT Hub, then into our Data Ingestion module which persists to TimescaleDB and fires a SensorDataReceived event. Step 3: the Data Quality Gateway validates — staleness, range, rate-of-change, compression artifacts — and detects the current operating mode. If mode is Normal, it fires ValidatedDataReady. Step 4: the Trigger Engine evaluates about 200 rules. Is this a meaningful process drift or price move? Has it persisted for at least 2 minutes? Has the 60-minute cooldown elapsed? If yes, it fires OptimizationTriggered. Step 5: the LP Orchestrator coalesces triggers within a 30-second window, dispatches a Celery task to the Windows LP Worker which runs Excel Solver via COM automation with a 5-minute timeout and watchdog. Step 6: here's the critical safety point — the AI Translation service programmatically extracts ALL numerical deltas from the LP output, stores them in a structured JSON column, then sends pre-validated numbers to Claude with a strict prompt template. Claude generates plain English text. We cross-validate every number in Claude's output against the source. If anything doesn't match, we fall back to a deterministic template. Finally, the Messaging Service delivers via Teams Adaptive Card — something like 'Naphtha yield +8% on Units 3 and 4 — plus $44K per day.' The supervisor reviews in 30 seconds and taps Acknowledge, Add Constraint, or Dismiss."

subtitle: "When a supervisor disagrees with a recommendation"
bullets:
- "5-tap structured wizard: unit -> type -> constraint -> severity -> confirm (15-30 seconds)"
- "Constraints stored by equipment ID, never by individual operator"
- "System never nags — conflicting recommendations suppressed until constraint expires or is cleared"
speaker_notes: "This is Reflex's core differentiator — the feedback loop. Say a recommendation comes in to increase throughput on Unit 2, but the supervisor knows heat exchanger 201 is fouling. They tap 'Add Constraint' on the Adaptive Card. A 5-tap wizard opens right in Teams: Tap 1 — select Unit 2. Tap 2 — select Equipment Limit. Tap 3 — select Heat Exchanger Fouling. Tap 4 — select severity, Cap at 75% capacity. Tap 5 — confirm. Total time: 15-30 seconds, fully glove-compatible with 64-pixel tap targets. This is 100% structured data — no NLP, no typing, no ambiguity. The constraint goes to the Constraint Registry, which stores it by equipment ID with a 72-hour default expiry. The LP Orchestrator immediately re-solves with Unit 2 capped at 75%. A revised recommendation comes back: 'With Unit 2 capped, increase diesel on Unit 6 instead — revised margin plus $11,200 per day.' The critical behavior: the system will NOT re-recommend Unit 2 until the constraint expires or is manually cleared. No nagging."

type: visual
title: "Network Architecture — OT/IT Boundary"
subtitle: "Purdue Model network segmentation"
bullets:
- "Edge Agent reads from read-only PI replica in DMZ — never touches OT systems"
- "ALL communication is outbound HTTPS only — no inbound firewall rules required"
- "Compatible with hardware data diodes (physically one-way)"
speaker_notes: "This is the slide that matters most to every refinery IT security team and CISO. Start at the left — the red zone is Purdue Level 0-2, the process control OT network. DCS controllers, PLCs, sensors, and the primary PI Data Archive all live here. Reflex never touches this network. Not at all. Ever. The PI historian replicates — using AVEVA's own managed process — to a read-only replica in the DMZ at Purdue Level 3.5. This is the orange zone in the middle. Our Edge Agent — a lightweight 60MB Python Docker container — reads from this read-only replica via PI Web API over HTTPS with Basic Auth. It buffers to local SQLite for store-and-forward resilience. The critical point: the Edge Agent communicates outbound only over HTTPS to Azure IoT Hub. There are zero inbound firewall rules. This is inherently compatible with hardware data diodes — physically one-way network devices. On the right in the blue-green zone is the Reflex Cloud on Azure. IoT Hub receives the sensor batches, feeds them into the FastAPI monolith, which reads and writes to PostgreSQL plus TimescaleDB. The two red firewall symbols between zones represent the security boundaries — OT to DMZ, and DMZ to IT/Cloud. This architecture means a compromise of the Reflex cloud cannot propagate back into the process control network."

title: "Infrastructure Decisions"
subtitle: "Cloud, compute, edge, and storage"
table:
  headers: ["Decision", "What We Chose", "Key Reason", "Risk It Addresses"]
  rows:
    - ["Cloud Platform", "Azure-primary with cloud-agnostic services", "Azure IoT Edge has first-party OPC-UA; PI Integrator is official AVEVA product; refineries are Microsoft 365 shops; $100/yr student credits + up to $150K via MS for Startups", "Vendor lock-in mitigated by using TimescaleDB (not Cosmos), Claude direct (not Azure OpenAI)"]
    - ["Architecture Pattern", "Modular monolith (Python/FastAPI) with one external Windows LP Worker", "Team of 3-5 students cannot manage N microservices x (logging + monitoring + deployment + versioning); clean module boundaries allow extraction later", "Operational complexity kills student teams; monolith is a feature not a bug at this scale"]
    - ["Edge Deployment", "Lightweight Python Docker (~60MB) + SQLite buffer, NOT Azure IoT Edge", "IoT Edge adds significant complexity for single-site pilot; core data flow doesn't need module orchestration; IoT Edge becomes the Phase 3+ migration target", "Over-engineering at MVP stage delays time-to-pilot"]
    - ["Database", "Single PostgreSQL 16 + TimescaleDB for ALL relational + time-series data", "One DB instead of two; full SQL JOINs between time-series and relational; standard PostgreSQL SQL; Azure student credits cover hosting", "InfluxDB would mean separate DB to manage with no JOINs — rejected in conflict resolution"]
speaker_notes: "Let me highlight the non-obvious decisions here. First, you might expect us to go cloud-agnostic or even multi-cloud from day one. We didn't — Azure wins on the single most important technical dimension for refineries, which is industrial IoT connectivity. But we hedged: TimescaleDB instead of Cosmos, Claude direct instead of Azure OpenAI, Terraform instead of Bicep. Second, modular monolith over microservices — this is often controversial, but for a student team, the operational overhead of managing multiple services would be a death sentence. We designed clean module boundaries so we can extract services later when we actually need to scale. Third, we chose a simple Python Docker container over Azure IoT Edge for the edge agent. IoT Edge is great at 10+ sites, but it's overkill for a single pilot. And fourth — single PostgreSQL with TimescaleDB instead of a separate InfluxDB. This was actually a conflict between our research agents that we had to resolve. The winning argument: one database is dramatically simpler to operate, and we need SQL JOINs between time-series sensor data and relational constraint data."

Many mid-size refineries use Aspen PIMS, Haverly GRTMPS, or Honeywell RPMS. Excel is often just the data I/O layer. That's why Phase 0 includes surveying 10-15 refineries — this is a binary market validation question that gates everything else. On the historian connector: our original plan was to go through Seeq, which is an enterprise analytics layer. Research showed it's $100K+ per year and completely overkill for what we need — basic data collection. PI Web API is REST-based and explicitly designed for DMZ access. On feedback — this is where we made the most radical change. The original concept had operators typing free text in Slack. Three problems: NLP accuracy is only 46-85%, operators wear chemical-resistant gloves that make typing impossible, and free text doesn't capture quantitative information. The 5-tap wizard solves all three: 100% accuracy, glove-compatible 64-pixel buttons, and predefined severity levels."

speaker_notes: "These are the decisions that protect us from destroying operator trust — which, in this market, you only get one chance at. Start with the LLM decision. You might expect Claude to process the LP output end-to-end. We deliberately chose NOT to do that. Research shows LLMs fabricate numerical errors in up to 83% of cases when errors exist in source material. In a refinery, confusing 2.3 MBPD with 23 MBPD is not a rounding error — it's a safety incident. So we programmatically extract every number, store them in structured JSON, send pre-validated numbers to Claude with strict instructions, and cross-validate every number in the output. If anything fails, we fall back to a deterministic template. About one week of extra engineering to eliminate the highest-impact failure mode. On dashboards — this is counterintuitive. You'd think showing operators how much money their overrides cost would motivate better decisions. Research says the opposite: loss framing decreases performance a third of the time. Worse, the United Steelworkers union is actively bargaining over AI in refineries. If operators feel surveilled, they either follow AI blindly — a safety risk — or disengage entirely. So we track by equipment, never by operator, and frame everything as value captured. On triggers — the original plan was a fixed $2 per barrel threshold. That's 8-20% of a normal spread but only 5% of an elevated spread. And during shutdowns — when 50% of safety incidents occur — nearly every sensor changes dramatically. We'd be firing meaningless recommendations when operators are under maximum cognitive load. So: percentage-based thresholds, operating mode gating, and targeting 1-2 recommendations per shift."
